# 产品闭环加固设计

## 目标

修复深度审核确认的真实缺口，并用当前源码、真实公开页面、模拟器和正式标识真机证据重新判断产品闭环状态。

## 范围

- QQ 音乐公开链接：基于真实页面边界提供可操作失败提示；分享载荷同时带有歌曲原文时继续从原文导入。
- Apple Music、网易云音乐回归：保留现有公开页面解析，不调用私有接口，并增加跨平台链接契约测试。
- 正式真机：使用 `com.huangwei.singreadyai`、正式分享扩展标识和 `group.com.huangwei.singreadyai` 构建、安装、启动；不把旧 `com.example` 包计入验收。
- 系统能力：分别验证分享扩展、照片选择/OCR、麦克风权限与录音、保存海报、系统分享和 Apple Music 跳转。不能自动完成的真实输入或第三方接收流程明确保留人工验收边界。
- 交付证据：源码稳定后重跑单元测试、完整 UI、Release 构建、截图矩阵和文档门禁，更新事实数字与结果路径。

## 方案

链接导入继续使用现有 `PlaylistPageTextExtractor` 分层：URL 安全策略负责限制公开 HTTPS 与重定向，平台解析器只从公开 HTML 中读取页面已下发的结构化数据，最后才回退到通用可见文本提取。真实探针确认 QQ 歌单入口连续跳转后只返回依赖 JavaScript 的空壳，公开 HTML 不含歌曲数据；因此不扩大 `198.18.0.0/15` 例外，也不调用 QQ 私有接口。URL-only 载荷直接返回 QQ 专用恢复提示；同一分享载荷若保留了可识别歌曲原文，则仍从原文继续整理。

现有任意域名抓取采用“预解析 DNS 后仍由 URLSession 再解析”的方式，无法消除 DNS rebinding 时间窗。为保持简单可靠的安全边界，生产网络抓取收紧到应用已有确定性解析器的 Apple Music 与网易云官方 host；QQ 在网络前分流，其他网页不发起请求并提示粘贴文字或截图。官方 host 的重定向仍逐跳限制在同一受信集合内，并由 HTTPS 系统证书校验确认服务身份。删除无法绑定到连接的 DNS 预检，不再把重复解析描述成 SSRF 防护；安全真源是固定官方 host、TLS 与逐跳重定向白名单。测试注入的页面 fetcher仍可验证通用文本解析器，但不再对用户输入的任意域名发网络请求。

真机验收以工程正式标识为真源。若 Apple 开发者凭据或 profile 阻止签名，先记录完整签名层证据并修复本机可修复配置；需要开发者门户权限的部分不伪造通过。

## 错误处理

- 私密、登录墙、动态页面无公开数据：按来源明确提示改用分享原文、粘贴文本或截图；QQ URL-only 不显示成网络故障。
- 代理虚拟地址：仅受信音乐 host 例外放行，重定向后的每个 host 重新验证。
- 系统权限拒绝：保留设置入口和不依赖该权限的替代路径。
- 正式签名失败：输出具体缺失的证书、App ID、App Group 或 profile，不降级成旧 Bundle ID 作为交付结果。

## 验证

每项生产代码先有可复现失败测试，再做最小修复。最终门禁包含 `swift test`、iOS 单元测试、107+ UI 回归、Release simulator/device 构建、正式真机安装启动、截图指纹与两套字号截图、文档事实校验和 `git diff --check`。
