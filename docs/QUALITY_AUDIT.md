# 质量审查

更新时间：2026-07-12

## 审查范围

- 以当前 live code、测试和文档核对产品功能分区、交互、数据来源、异步边界、恢复、删除、可访问性、代码结构、性能预算和可重复验证能力。
- 早期 11 个关键状态截图只作为历史对照；当前最终源码已重新生成常规与最大辅助字号各 11 张截图，并完成完整 UI 回归和人工复查。
- 修复后重复验证，并记录后续审查中再次发现的问题和处理结果。

## 初始发现

- P0：Share Extension `Info.plist` 缺少真实 `NSExtension` 注册信息。
- P0：App 录音权限用途文案缺失。
- P0：录音链路只做计时和音量展示，最终仍使用默认示例范围。
- P1：设计系统散落在页面文件中，缺少独立 token 和组件层。
- P1：`ProductFlowViews.swift` 超过 1000 行，页面、状态和组件耦合。
- P1：旧四 Tab 页面仍在 target 中，视觉和文案与新流程冲突。
- P1：缺少真实截图证据和 UI test 覆盖。
- P1：文档声称的验证结果与当前代码不一致。

## 修复结果

- 补齐 Share Extension activation rule、principal class、display name 和 App Group 配置。
- App target 补齐麦克风隐私用途文案。
- 新增 `VoiceRecordingService` 和 `AudioFrameSplitter`，真机使用 `AVAudioEngine` 收集 10 秒声线样本并进入 `PitchDetector` 分析；赶时间时可先按常见音域排歌。
- 拆出 `SingReadyAI/App/DesignSystem`，包含颜色、字体、间距、动效、组件、无障碍和材质 token。
- 删除旧四 Tab 页面，按页面职责拆分 Onboarding、Import、Match/Voice/Scenario、Result/Export/Start Tips。
- 新增 UI test target 与 `scripts/capture_ui_test_screenshots.sh`，覆盖 onboarding、首页、导入、整理、匹配、声线、场景、歌单、导出、开唱小抄 11 个状态。
- 新增本地质量门禁脚本，覆盖 Swift tests、fixture、plist/privacy、docs、design、performance、screenshots、可选项目生成和通用模拟器构建。

## 修复中发现并解决的回归

- Onboarding 标题在真实窗口截图中被底部按钮遮挡：改为明确分区布局、自定义分页点和滑动手势。
- 导出页从指定功能启动时继承上一页长列表滚动位置，导致标题离屏：先通过页面身份重建修复；后续改为原生 `NavigationStack` 后，每个功能页由独立导航目标承载，滚动状态不再跨功能继承。
- Onboarding 修复后主按钮底部仍接近安全区边缘：压缩中段视觉元素并使用底部 safe-area padding。

## 二次审查发现并解决的问题

- 首页“示例歌单”入口视觉权重过大，和旁边导入入口不像同一套组件：改为同级功能卡片，并统一圆角和内边距。
- 导入页入口卡片和背后容器圆角不一致：统一使用设计系统里的标准圆角。
- 产品被包装成长流程，用户像是必须顺着 8 个环节走完：改为首页按“我有歌单 / 唱前准备 / 到了现场”分区，顶部菜单可从任意页面直接切换功能。
- Onboarding、声线页存在“跳过”“下一页”等流程感过强的文案：改为“开始使用”“看看怎么用”“先不测”，让用户按当下需求选择。
- KTV 匹配 100% 作为主结论显得机械：改成“都能唱 / 多数 / 一半多 / 先挑”等判断词，明细也不再用裸百分比压用户。
- 大字号下顶部功能菜单文字会被压成省略号：无障碍字号下切换为图标入口，并保留可访问性标签。
- 部分说明文案偏内部表达：清理“先跳过”“待导入”等不贴近真实用户场景的表达。

## 三次审查发现并解决的问题

- App 仍容易被理解成必须按顺序走完的长流程：继续强化首页“先做哪件事”的功能分区，顶部菜单按“我有歌单 / 唱前准备 / 到了现场”直达各功能。
- 整理页每首歌卡片过高，用户扫 12 首歌负担重：改成紧凑编辑行，默认只保留歌名、歌手和删除，不确定条目才显示说明。
- 满匹配时仍显示“再备几首 / 还没加备选歌”，造成没完成的感觉：仅在存在模糊、替代或未匹配歌曲时展示备选歌模块。
- 声线页“够不够用 / 够参考”等表达不够自然：改为“这次结果 / 可以参考”，并补充“怎么测都行”的普通用户提示。
- 场景页主按钮出现在选项前，像在催用户跳过选择：把“排一份今晚歌单”放到场景和偏好选项之后。
- 结果页每首歌默认露出“搜索 / 复制 / 锁定 / 详情”，工具感过重：默认只保留“锁定 / 详情”，搜索和复制收进详情。
- 导出页把“复制歌单”压到海报下方，首屏动作不完整：首屏改为“存海报 / 复制 / 分享”，可访问性仍保留完整语义。
- 无障碍大字号下标签和主按钮出现省略号：标签改为大字号单列，主次按钮允许两行并自动增高。
- 匹配、场景和歌单页仍有评分表味道：把“适合度 / 难度 / 高音多不多 / 合唱好不好接”等指标改成“很适合 / 适中 / 还好 / 很少”等自然判断。
- `DesignSystemComponents.swift` 和 `MatchVoiceScenarioViews.swift` 再次逼近长文件：拆出 `DesignSystemMetrics.swift` 与 `VoiceAndPreferenceViews.swift`，并让设计校验脚本扫描整个 DesignSystem 目录。

## 四次审查发现并解决的问题

- 顶部栏原副标题像口号，信息价值低：改为显示当前功能名，例如“导入歌单”“整理歌单”“排今晚歌单”。
- 导入页两个动作偏内部操作语气：改为“用示例歌单 / 整理这段歌单”。
- 声线页部分说法偏术语：改为“哪段更舒服 / 先不测”，并同步录音失败、隐私提示和手动验收文档。
- 场景页标题和部分卡片提示拗口：改为“按今晚的局来排 / 先唱大家熟的 / 轻松一点”等更自然的 K 歌场景文案。
- 开唱小抄卡片标题不够像正常中文标题：改为“开场别冷场 / 顺序好接歌”。
- 文档仍残留旧叫法：同步 `START_TIPS_SCRIPT.md`、`MANUAL_QA.md`、`GAP_ANALYSIS.md`、`PERFORMANCE_BUDGET.md` 和 UI 测试。

## 五次审查发现并解决的问题

- 首页虽然已经分区，但仍有轻微清单感：把主标题下的提示改为“先做哪件事”，强化导入、测音域、排歌、分享和开唱小抄都能按需打开。
- 导入、结果和分享页部分文案仍偏报告或工具口吻：改为“选一种方式”“发群里更省事”“朋友局先唱大家熟的”等更像普通用户会说的话。
- 推荐摘要和复制分享文本仍会带出长句：共享层同步改写 `PreferenceProfiler`、`RecommendationEngine` 和 `PlaylistTextExporter`，避免 UI 短文案和分享文案不一致。
- 结果、声线建议或测试夹具里仍可能露出偏术语的唱歌说法：用户可见文案改为“想唱”“舒服范围”“备选歌”等更直接的说法。
- `accessibility-extra-large` 下匹配圆环会把“都能唱”截成省略号：无障碍字号下圆环改为符号显示，完整结论保留在页面标题和可访问性标签里。

## 六次审查发现并解决的问题

- iPhone Air 真机安装后出现非全屏/尺寸异常的风险：项目之前声明 iPhone+iPad 设备族，但当前视觉和测试都按 iPhone 交付；本轮收窄为 iPhone-only，并移除未验证的 iPad 方向声明。
- iOS 26 玻璃质感覆盖不完整：主 App 的 `LiquidGlassSurface` 在降级路径补齐同形圆角背景，分享扩展也增加 iOS 26 glass 和 Reduce Transparency 降级。
- 顶部菜单“换一项”仍像流程切换：改成更普通的“功能”，可访问性标签改为“打开功能菜单”。
- 首页和声线页仍残留工具说明口吻：把“按需要选择 / 直接排歌 / 默认范围 / 参考程度”改为“想做什么 / 先不测 / 常见音域 / 这次结果”等更贴近 C 端的说法。
- 分享扩展虽然功能闭环，但视觉像单独弹窗：补齐标题图标面、卡片面和按钮面的统一半径、边线和玻璃材质。
- 无障碍大字号下 Hero 眉标会和顶部当前功能名重复：大字号时隐藏眉标，保留主标题和顶部功能名。
- 导出页和小抄页仍残留“开唱提示”旧叫法：统一成“看开唱小抄 / 开唱小抄”，更像用户会点开的现场工具。

## 七次审查发现并解决的问题

- 首页和顶部菜单仍有“把长路径拆成卡片”的残留：改为“我有歌单 / 唱前准备 / 到了现场”，更像用户按当前需求选工具。
- 从首页或顶部菜单进入后续功能时，旧逻辑会自动塞一份示例歌单，容易让用户误以为必须走完前面环节：普通跳转现在只打开对应模块，由模块自己的空状态告诉用户需要什么内容。
- Onboarding 第一屏仍暗示“先整理、再匹配”的顺序：改为“想整理歌名、查 KTV、排歌都能直接做”，并补充“想测就测，不想测也能排”。
- 场景页说明还像配置项提示：改成“不想细调就直接排；想换场合、人数或氛围，再点下面的选项”。
- UI 测试新增首页直达空状态、onboarding 浏览说明和开唱小抄长内容滚动覆盖，避免后续改动把独立模块重新变成长流程。

## 八次审查发现并解决的问题

- 真机冷启动仍显示系统占位图标：补齐 AppIcon 资产目录、项目资源引用和编译设置，并验证构建产物中的图标与 `CFBundleIconName`。
- 结果页和网格按钮看起来能点，但辅助功能读取到的点击区域不足 44pt：统一在共享按钮样式补足最小点击区域，并增加实际 frame 的 UI 回归测试。
- 分享扩展在共享存储失败时提示复制，却没有复制按钮：增加“复制内容 / 已复制”操作，并修复截图文件在异步回调结束后失效的风险；截图失败则明确要求回到 App 重新选择。
- 导出页声称适合发群，但系统分享面板里仍是长理由和搜索链接：新增精简分享格式，只保留场景、分段和歌名。
- 超大字号引导页正文会被省略：改成完整文本和可滚动内容区。
- 匹配、声线和车载场景仍有几句偏说明书：改成“把歌单放进来”“不用唱完整”“难唱的别连着排”等正常口语。

## 九次审查发现并解决的问题

- 旧版根视图直接替换页面，没有原生返回历史，也不能可靠使用 iOS 左缘滑动返回：改为带路径的 `NavigationStack`，流程内前进保留上一页，全局功能菜单切换则重置到首页层级。
- 顶部双层自定义栏占高且层级重复：移除自定义返回栏，交给系统导航栏承载标题、返回和功能菜单；iOS 26 隐藏栏背景，让系统液态玻璃按钮直接呈现，iOS 17 至 iOS 18 使用系统材质降级。
- 冷启动指定功能时可能连续触发中间页面跳转：数据准备和导航拆开，启动只在准备完成后一次性设置目标路径。
- 录音权限请求或录音过程中离开页面，异步任务可能把用户重新带回声线页：页面离开时取消录音任务，取消状态回到待开始，不显示错误也不触发跳转。
- 锁定或反馈后重新生成歌单会创建新卡片身份，导致详情收起和滚动跳动：相同场景和歌曲保留稳定标识，展开状态和阅读位置不再被刷新打断。
- 空歌单、备用歌曲和麦克风进度仍有实现口吻：改成“歌单空了，请至少留一首歌”“刚才没搜到更多”“正在打开麦克风”等面向用户的表达，并加入文案回归检查。

## 产品闭环深审发现并解决的问题

- 215 首本地数据此前容易被理解为真实 KTV 可点曲库：统一改为“本地参考曲库/常见 K 歌参考”，并明确它是 early-validation fixture，不代表平台、地区、门店或设备的实时库存。
- 匹配页此前只有汇总，缺歌手的同名歌曲可能被直接当作精确命中：增加逐歌核对、待确认状态和采用备选操作；用户确认前不进入画像和参考匹配率。
- 常见音域兜底曾与用户实测混用：增加声线来源，UI、推荐理由、调性建议和导出分别处理“本次唱到的音区”和“常见音域参考”。
- 时长、锁定、移除和反馈控制存在语义冲突：计划数量改为按时长确定，锁定跨场景保留，锁定超过目标时扩容提示，反馈维度可并存，场景硬规则只替换不额外追加。
- 外部候选曾被写成“相似歌曲”：当前运行时明确为 Apple 公开搜索的同歌手曲目，未知的 KTV 收录、音域、难度和现场适配不参与确定性结论；请求增加超时、取消和歌单内容身份校验。
- 链接能力容易被域名识别夸大：生产网络现只读取 Apple Music 与网易云官方公开链接，并逐跳限制官方 HTTPS 主机；QQ 音乐纯链接和其他网页直接进入文本/截图恢复路径。仅当分享载荷确实保留可识别歌名原文时才做本地文本解析。
- Share Extension 共享容器失败时，扩展自己的 Application Support 不能作为主 App 可读的后备存储：链接/文本改为手动复制，截图改为回到 App 重新选择。
- 导入、录音和候选请求此前存在迟到提交或旧计划继续可用的风险：导入离页/取消/超时会使请求 token 失效，首次麦克风授权不再被 inactive 状态误取消；整理内容、场景、声线和候选变化会废弃旧计划。
- 空歌名此前可进入匹配：parser、整理页和提交边界现均阻止空白标题。
- 移除歌曲此前只有单步内存撤销：结果页增加可持久化的已移除歌曲管理，支持逐首和全部恢复。
- 独自练歌、情侣和车载小抄此前复用多人聚会文案：现按场景生成全部标题、标签、歌曲提示、换歌和分享说明，车载明确驾驶者不操作手机。
- 截图此前只限制文件体积：现先读元数据限制像素与边长，OCR 使用有界缩略图，并在启动/全清时清理临时图片。
- Share provider 此前可能永不回调：扩展提供显式取消与总体读取时限，超时会取消 Progress、清理暂存并进入手动恢复态。
- Share provider 的 URL 表示此前会串行阻塞纯文本：现并行读取各表示并持续保存已完成部分，总超时后仍可复制先到的文本。
- 已导入但未核对时直接排歌此前会用热门歌单覆盖当前上下文：现只在没有导入内容时使用热门兜底，有当前歌单时先完成合法匹配。
- 确认本地匹配或采用替代歌此前会先清空外部候选：现只取消在途请求，保留已加载候选并使旧计划失效。
- 待处理分享此前可能在最近记录保存失败后仍被删除：现同步尝试最近记录与当前快照，至少一处成功才消费；双失败保留原项和截图。
- 当前进度此前不能可靠跨重启恢复：增加版本化 `WorkflowSnapshot`，保存整理、匹配、声线来源、场景、计划、锁定、移除、外部候选和反馈；恢复时会丢弃与已编辑歌单或新场景冲突的旧下游结果。
- 最近导入此前保留原始版本且缺少数据管理：整理结果按 playlist ID 覆盖回写，待处理项和最近导入可单删，并提供清除待处理、最近导入、当前计划、反馈、共享截图和隔离文件的统一入口。
- 导出页此前只有精简分享文本：现在保留群聊短格式，并增加真实可分享的 UTF-8 详细 `.txt` 文件。

## 最终深审发现并解决的问题

- 独立保存的歌曲反馈可能比工作流快照更新，但恢复后的旧计划仍保留旧标签和顺序：恢复时比较反馈真相，必要时自动重排；缺少生成条件时直接使旧计划失效。
- 旧版本只有快照内实测音区、尚无独立音区文件时，清理工作流快照后可能丢失测量：恢复时把有效实测迁移到独立本机存储，并保持清除世代保护。
- 相同 Toast 文案连续出现时不会重置生命周期，旧任务还可能提前清除新提示：每次展示改用独立身份，关闭任务只能清理自己的提示；长 warning 在常规与辅助字号下均支持多行，success 保持紧凑。
- Release 产物门禁此前只检查隐私清单，未证明离线隐私政策确实打包且与根文档一致：新增产物逐字校验，并补齐正常字号和最大辅助字号的整页滚动、正文链接和原生返回 UI 测试。
- 匹配页外部候选提示曾误写成向 Apple 发送“歌名和歌手”，与真实请求和隐私政策的“只发送歌手名称”冲突：补充字段级 UI 合同，修复前聚焦测试稳定失败，改正披露后通过。
