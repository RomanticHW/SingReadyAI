# 产品闭环修复设计

## 目标

在不接入私有音乐接口、真实 KTV 厂商协议、账号系统或云同步的前提下，修正当前版本中会让用户误解数据来源、个性化程度和功能完成度的问题，使 App 达到可信、可恢复、可重复验证的 early-validation 状态。

## 设计原则

1. 本地样例只能表达为参考，不能表达成特定门店真实可点。
2. 常见音域兜底与用户实测必须是两种可区分的数据来源。
3. 外部歌曲只有候选元数据时，未知的音域、难度和 KTV 可用性不得参与确定性结论。
4. 用户可见控制必须兑现：时长要改变数量，锁定不能静默失效，反馈维度应互不覆盖。
5. “提前准备、到店使用”要求当前计划可跨重启恢复，用户也必须能删除本地内容。
6. 当前阶段只补真实可达能力；不以占位实现冒充厂商曲库、真实平台解析或现场协作。

## 一、数据来源与表达

### 本地参考曲库

- 保留 215 首本地曲库用于交互和推荐验证。
- 用户界面统一称为“本地参考曲库”“常见 K 歌参考”或“常见度参考”。
- 不再使用“门店能点到”“能直接唱”等没有厂商和地区依据的承诺。
- 匹配页显示每一首导入歌曲、匹配状态、命中歌曲、匹配原因和备选项。
- 同名但缺少歌手的歌曲不能直接判为精确匹配，即使只有一个候选也必须让用户确认。
- 匹配结果区分“待确认”和“已确认”；待确认候选不能进入画像、推荐或匹配率，用户确认后才重建画像并保存。

### 推荐输入来源

- 推荐计划记录歌单来源：用户导入、示例歌单或热门兜底。
- 声线记录来源：用户实测或常见音域参考。
- 只有用户真实导入时，推荐理由才能使用“你歌单里的歌手”等表达。
- 只有用户实测且结果有效时，才能出现“本次音区与歌曲接近”等声线理由。
- 常见音域不展示性别、实测置信度或“你的音域”。

## 二、声线测量与调性建议

- 引导用户从舒服低音逐步唱到舒服高音，结果称为“本次唱到的音区”。
- 用户可见信息使用音高范围和宽窄，不根据一次录音输出男声/女声类型。
- 音高样本跨度不足时返回可操作的重试提示，不生成精确结论。
- 调性建议同时考虑歌曲高低音范围；常见音域和外部候选不生成原调或精确降调结论。
- 录音过程中只允许取消，不允许重复开始或直接切换到兜底结果。
- 离页、进后台和取消都必须结束录音任务并关闭音频会话。

## 三、推荐语义

- 基础目标数使用 `min(30, max(6, ceil(时长分钟 / 5)))`，30、45、60、90、120、180 分钟分别对应 6、9、12、18、24、30 首。
- 锁定歌曲优先于基础目标；有效目标数为 `min(唯一候选数, max(基础目标数, 可用锁定数))`。锁定数超过基础目标时全部保留，并显示“锁定歌曲使计划超出时长建议”。
- 候选不足时返回全部唯一候选并明确提示；各段容量合计必须等于实际输出数，场景硬规则通过替换实现，不能在容量外追加。
- 锁定歌曲跨场景仍保留；若与场景冲突，保留并显示风险，而不是静默移除。锁定与移除冲突时锁定优先。
- 喜欢、太高、不熟、唱过、适合合唱属于不同维度，可以同时存在。
- 独自练歌固定一人，情侣局固定两人；其他场景允许合理人数范围。

## 四、外部候选

- 当前运行时使用 iTunes 同歌手搜索，因此界面称为“同歌手备选”，不声称算法相似度。
- 外部候选保留来源和待确认标记；未知音域、难度、可用性不参与评分、排序、风险提示或精确适配结论。
- provisional 候选的排序只使用公开元数据相关度、与导入歌手/歌名的关系和用户显式反馈；修改内部占位数值不得改变其排名或理由。
- 以规范化歌名和歌手去重，本地参考曲优先于外部重复项。
- 推荐候选池必须显式保留外部候选和锁定歌曲，不因本地曲库数量上限被截掉。
- 外部请求带总时限、取消和歌单身份校验；旧歌单请求不得写入新歌单。
- 点击前说明会把少量歌名或歌手发送到公开搜索服务，仅用于候选查询。

## 五、恢复与数据管理

- 使用版本化 `WorkflowSnapshot` 保存当前导入上下文、匹配、画像、声线来源、场景、计划、锁定、移除和外部候选。
- 整理中的编辑和删除，以及确认匹配、锁定、移除、反馈、场景变化和外部请求完成都触发快照更新。
- 冷启动恢复后，首页可直接继续当前歌单、导出或开唱小抄。
- 整理后的歌名、歌手和删除结果按 playlist id 覆盖最近导入，不能恢复原始错误版本。
- 提供待处理项单删、最近导入单删以及“清除本机记录”。
- 清除操作同时移除共享截图、最近歌单、当前计划、本地反馈、隔离文件和孤立截图。
- 主 App 每次回到前台时刷新 Share Extension 队列。

## 六、导入与导出

- Share Extension 合并同一次分享中的 URL 与纯文本，不因先发现 URL 而丢弃原文。
- 含 URL 的分享文本同时保留 URL 与原文；仅当同一分享载荷确实保留了原文时，公开网页读取失败后才继续尝试解析其中的歌名。
- 纯文本没有解析出歌曲时显示错误，不进入空整理页。
- 生产链接读取限定为 Apple Music 与网易云官方公开页面，并逐跳约束官方 HTTPS 主机；QQ 音乐及其他网页进入分享原文/截图恢复路径，不把来源识别等同于平台歌单解析。
- Share Extension 无法写入 App Group 时不使用扩展自己的本地目录冒充跨进程后备存储；链接或文本提供手动复制，截图要求用户回到主 App 重新选择。
- 保留群聊精简分享，同时提供真实的详细 `.txt` 文件分享。
- 文档只声明 UI 可达且经过测试的能力。

## 七、验证策略

- 所有行为变更先添加失败测试，再做最小实现。
- SharedKit 单元测试覆盖来源解码兼容、匹配歧义、推荐数量、锁定、反馈、外部去重、存储恢复和文本回退。
- UI 测试覆盖逐歌匹配、常见音域表达、录音取消、当前计划重启恢复、本地数据删除和详细文件入口。
- 可注入单元测试覆盖 scene inactive 停录、外部超时与旧请求拒写；匹配测试覆盖单候选待确认、多候选待确认、确认后重建和快照恢复。
- 最终运行 `swift test`、聚焦 UI 测试、完整 UI 测试、通用模拟器构建和 `./scripts/validate.sh`。
- 修复后重新按产品声明、运行路径、异常状态和持久化边界做只读深审。

## 不在本轮范围

- 真实 KTV 厂商曲库、门店库存和设备控制。
- 音乐平台私有接口、版权音频、歌词、MV 或封面。
- 账号、云同步、多人协作、iPad 专用布局。
- 完整的现场点歌队列与播放控制；若深审后仍值得做，再进入独立设计。
